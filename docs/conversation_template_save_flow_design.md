# /app/conversation 템플릿 저장 UX 개편 설계

## 목표
- 템플릿 리스트/정책 카드에 분산된 저장 버튼을 제거하고, 변경 사항이 있을 때만 노출되는 하단 고정 저장 바(팝업)를 제공한다.
- 저장 동작을 단일 진입점으로 통합해 사용자가 저장 시점을 명확히 인지하도록 한다.
- 기존 저장 API 계약과 데이터 저장 흐름은 유지하되 UI/UX만 통합한다.

## 변경 배경
- 현재 저장 버튼이 템플릿 리스트(아이템별)와 정책 카드에 분산되어 있어 저장 타이밍이 불명확하다.
- 변경 사항이 존재할 때만 명시적으로 저장을 유도하는 UI가 필요하다.

## 현행 동작 (코드 확인)
- `/app/conversation` 페이지는 `src/app/app/conversation/page.tsx`에서 렌더링된다.
- 템플릿 리스트에서 선택된 행에는 `Pencil/Save` 아이콘 버튼이 표시되며, `isDirty`일 때 저장 아이콘으로 전환된다.
- 상단 정책 카드(“대화 정책”) 우측에 `정책 저장` 버튼이 있으며 `handlePolicySave()`를 호출한다.
- 리스트 저장 버튼도 `handleSave()` → `handlePolicySave()`로 동일 저장 로직을 호출한다.
- `handlePolicySave()`는 템플릿 PATCH(`/api/widget-templates/:id`)와 정책 POST(`/api/widget-templates/:id/chat-policy`)를 순차 실행한다.

## 변경 요구사항 요약
- 템플릿 리스트의 수정/저장 버튼(연필/저장 아이콘)을 제거한다.
- 정책 카드의 `정책 저장` 버튼을 제거한다.
- 변경 사항이 있을 때만 화면 하단에 고정된 `저장하기` 버튼(바/팝업 형태)을 표시한다.
- 변경 사항이 없으면 하단 저장 바는 숨김 상태를 유지한다.
- 저장 버튼 UI는 1차 생성 후 사용자에게 원하는 형태를 재확인하는 절차를 반드시 거친다.
- 저장 버튼/바 UI는 `src/components` 내 중앙화된 디자인 요소를 최대한 활용한다.

## 설계 개요
- `isDirty`를 저장 바 노출의 단일 기준으로 사용한다.
- 저장 바는 뷰포트 하단에 고정(`position: fixed`)되며 스크롤과 무관하게 항상 보인다.
- 저장 바는 “변경 사항 있음” 상태를 인지할 수 있는 텍스트(예: “변경 사항이 있습니다”)와 단일 `저장하기` 버튼을 포함한다.
- 저장 바는 `isDirty`가 `false`가 되면 즉시 숨김 처리한다.
- 저장 버튼 클릭 시 기존 `handlePolicySave()`를 호출한다.

## UI/UX 동작 상세
- 노출 조건: `isDirty === true`.
- 숨김 조건: `isDirty === false` 또는 `draft`가 없을 때.
- 위치/형태: 페이지 하단 고정, 너비는 `max-w-6xl` 기준 중앙 정렬, 카드형 배경.
- 접근성: 버튼은 `type="button"` 사용, `aria-label="저장하기"` 제공.
- 저장 중 상태: 기존 `policySaving` 상태를 반영해 버튼 텍스트를 `저장 중...`으로 변경하고 `disabled` 처리.

## 상태/이벤트 모델
- `isDirty`: 템플릿 기본 정보/도메인/경로/테마/설정 변경 여부를 판단하는 기존 계산식을 유지한다.
- `policyValue`: ChatSettingsPanel 변경 시 `onChange`로 갱신된다.
- 저장 이벤트: `handlePolicySave()` 단일 호출로 템플릿 PATCH + 정책 POST 수행.
- 저장 성공 후 기대 상태: 저장 완료 시 `isDirty`가 `false`로 내려가며 저장 바가 자동 숨김.

## 제거 대상 UI
- 템플릿 리스트 우측 액션 영역의 수정/저장 버튼(연필/저장 아이콘).
- 정책 카드 상단의 `정책 저장` 버튼.

## 구현 메모 (의도)
- 저장 바는 `src/app/app/conversation/page.tsx` 내부에만 추가하여 변경 범위를 최소화한다.
- 기존 저장 로직과 API 호출은 변경하지 않는다(계약 유지).

## 테스트 시나리오 (요구 수준)
- 템플릿을 선택 후 이름/정책/설정 변경 → 저장 바 노출 확인.
- 저장 바의 `저장하기` 클릭 → 템플릿/정책 저장 성공, 저장 바 숨김 확인.
- 변경 사항 없는 상태 → 저장 바 미노출 확인.
- 저장 중(`policySaving`) → 버튼 disabled 및 `저장 중...` 표시 확인.

## 실행 정책 (필수 준수)
아래 정책은 본 설계 또는 후속 수정에서 100% 준수한다. 간결하게 요약하지 않고, 실제 실행 단계에서 누락이 없도록 상세하게 기록한다.
- 작업 시작 전, `src/app/app/conversation/page.tsx`의 기존 저장 버튼 위치와 이벤트 바인딩을 재확인한다.
- 저장 흐름 변경이 계약/정책 레벨 변경인지 확인한다. 이 작업은 UI 통합이며 저장 API 계약은 변경하지 않는다.
- 변경된 UI가 기존 저장 로직(`handlePolicySave`)과 동일하게 동작하는지 코드를 통해 확인한다.
- 하단 저장 바는 `isDirty` 조건만으로 노출/숨김을 제어하고, 임시 플래그나 케이스별 예외를 추가하지 않는다.
- 저장 버튼 제거는 템플릿 리스트와 정책 카드에서만 수행하고, 저장 로직 함수는 제거하지 않는다.
- 구현 중 새로운 상태를 추가해야 한다면, 기존 상태(`isDirty`, `policySaving`)와 충돌하지 않는지 확인한다.
- 저장 버튼의 시각적 형태는 1차 구현 후 사용자에게 재확인 요청을 반드시 수행한다. 사용자가 승인하기 전에는 최종 확정으로 간주하지 않는다.
- 저장 버튼/바는 `src/components`에 정의된 중앙화 UI 컴포넌트(예: Button, Card 등)를 최대한 활용하고, 중복 스타일 정의를 지양한다.
- 구현 완료 후 `npm run build`를 로컬에서 실행한다. 실행할 수 없는 경우, 실행 불가 사유를 명시한다.
- 동작 검증은 반드시 `chrome-devtools` MCP로 `/app/conversation` 페이지에서 직접 확인한다.
- 동작 검증은 반드시 `supabase` MCP로 저장 API가 정상 호출/반영되는지 확인한다.
- 테스트 결과는 문서 하단 체크리스트 및 테스트 기록에 남긴다.
