# 위젯 로드시 초기 설정 불일치 현상 개선 설계

작성일: 2026-03-08  
작성자: Codex  
대상: `/embed/[key]` 위젯 로딩 과정

## 문제 요약
위젯 로드시 다음 설정이 즉시 반영되지 않고, 잠시 기본값으로 렌더링되는 현상이 발생한다.
- `widget.defaultTab` 설정
- URL 파라미터 `tab=policy` 등 강제 탭 지정
- `/app/conversation`에서 설정한 on/off 항목

이로 인해 사용자가 보는 초기 화면이 **의도된 탭/기능 상태와 다르게 짧게 표시**된다.

## 원인 가설 (기술적)
1. **설정이 비동기 로딩**되는 동안 UI가 기본값으로 먼저 렌더됨
2. `widget_config` 수신 이전에 `defaultTab`/`tab`/`pageFeatures`가 **초기 값으로 계산**
3. `tab=policy` 등 강제 파라미터가 있어도, 초기 렌더 시점에 `allowedTabs`가 아직 확정되지 않아 임시 탭 선택이 발생

## 개선 목표
- 초기 렌더링부터 **설정과 일치하는 UI만 표시**
- 비동기 로딩 중에는 **중간 상태(잘못된 탭/기능 표시)를 숨김**
- 탭/기능 상태가 확정되기 전에는 **UI를 잠시 보류**하거나 **로딩 스켈레톤** 표시

## 해결 전략 (설계)

### 1) “설정 준비 완료 플래그” 도입
위젯 내부에서 다음을 만족할 때만 “실 UI”를 렌더링:
- `widget_config` 수신 완료
- `pageFeatures`/`setupUi` 계산 완료
- `allowedTabs` 확정 완료

**준비 전에는** 임시 화면(로딩 스켈레톤 또는 빈 화면)으로 유지.

### 2) 초기 탭 결정 로직 2단계화
- 1단계: 설정 로딩 전에는 `activeTab`을 **결정하지 않음** (또는 `null`)
- 2단계: 설정 로딩 완료 후 `tab` 파라미터 / `widget.defaultTab` / `allowedTabs` 순으로 **한 번만 확정**

### 3) 정책/기능 토글 반영 타이밍 고정
`pageFeatures`/`setupUi`가 확정되기 전에는 다음을 숨김:
- 탭바
- 개별 패널 (`chat/list/policy/login`)

즉, “초기 화면”은 **정책 확정 이후에만 노출**한다.

### 4) 요청 파라미터 우선순위 명시
탭 선정 우선순위를 고정:
1. `tab=` URL 파라미터
2. `widget.defaultTab`
3. `allowedTabs[0]` (fallback)

이 우선순위는 설정 로딩 완료 이후 **단 한 번만 실행**되도록 고정한다.

## 적용 파일 (예상)
- `src/app/embed/[key]/page.tsx`  
  - 설정 준비 상태 관리
  - 탭/기능 렌더 지연
  - 초기 탭 선정 2단계화

## 비고
이 설계는 “잘못된 탭이 잠깐 보였다가 바뀌는 현상”을 원천적으로 제거하기 위해,  
**설정 확정 전에는 UI를 노출하지 않는다**는 원칙을 따른다.
