# intentupdate 설계 구현 체크 및 장애 기록

- 작성일: 2026-03-12
- 기준 설계: `docs/intentupdate.md`
- 근거 로그: `docs/logs/servicePageLog(bot-20260312-0847).md`, `docs/logs/servicePageLog(bot-20260312-0911).md`, 2026-03-12 Chrome-devtools 테스트, Supabase `F_audit_events`(session_id=`aa6c2056-7bd4-46d1-a0ae-9133d2734ec5`)


## 1. 핵심 사항 및 구현 저해 문제

1. 의도 분류 후 보존된 처리 흐름(재입고/배송지 변경 등) 실행
핵심 사항: 사용자 발화에서 의도를 분류하고, 재입고/배송지 변경 등 이미 설계된 런타임이 해당될 경우 그 흐름을 파괴하지 않고 그대로 실행해야 한다.
구현 저해 문제: 비로그인 위젯(/embed)에서 요청 실패 또는 토큰 오류가 발생하여 해당 흐름의 실제 동작 여부를 검증하기 어려웠다. 2026-03-12 테스트에서는 “볼캡 재입고”가 정상 흐름으로 동작했으나, 과거 로그 구간은 **검증 불가 상태**가 지속되었다.

2. 지원되지 않는 의도는 KB 기반 LLM 답변으로 처리
핵심 사항: 지원 흐름이 없는 의도는 KB를 읽고 LLM이 답변한다.
구현 저해 문제: 위젯의 KB 입력란에 실제 KB가 있어도 “적용” 버튼이 비활성 상태이며, 런타임 로그에서는 `kb_topics=["재입고"]`처럼 샘플 KB만 참조되는 정황이 반복되었다. 즉 **사용자 KB가 런타임에 반영되지 않음**이 가장 큰 장애였다. 2026-03-12 테스트에서는 `inline_kb`가 전달되면 정상적으로 KB 응답이 생성됨이 확인되어, 핵심 문제는 **입력 전달/동기화 실패**로 확정되었다.

3. KB 근거가 없으면 “KB에 없음”을 명시하고 KB 범주 내 재질문
핵심 사항: KB에 근거가 없으면 추정 답변을 금지하고, KB 범주 내 재질문을 한다.
구현 저해 문제: 실제 KB에 “신청기간/신청방법/제출서류”가 있음에도 `KB_NO_MATCH`로 처리되며 “KB에 없음”만 반복되는 로그가 확인됐다. 근본 원인은 **KB가 잘못 전달되거나 inline KB로 고정되는 문제**로 보이며, 정상 근거 탐색 자체가 불가능했다.

4. KB 매칭은 키 + 본문 기반으로 일반화
핵심 사항: 키 매칭 실패 시에도 본문 기반 매칭으로 근거를 찾는다.
구현 저해 문제: `KB_MATCH_COMPLETED`에서 `matched=false`가 계속 기록되고 `evidence_keys=[]`로 남는다. 본문 매칭 개선 전/후 여부와 관계없이, **실제 전달된 KB 내용이 재입고 샘플로 고정되어 매칭 대상이 엇나감**이 확인됐다.

5. LLM 프롬프트 계약: KB만 사용, 근거 키 기록
핵심 사항: LLM이 KB 외 정보를 사용하지 못하도록 프롬프트 계약을 고정하고, 근거 키를 로그에 기록한다.
구현 저해 문제: `KB_LLM_PROMPT_BUILT`에 `evidence_keys=[]`, `reason=KB_NO_MATCH`가 반복 기록되고 `kb_llm_fallback`으로 종료된다. 즉 **근거 키 자체가 생성되지 않는 상태**로, 계약을 지켜도 정상 답변을 낼 수 없었다.

6. pending 상태가 있으면 기존 처리 흐름 유지
핵심 사항: pending 상태(재입고/배송지 변경 등) 존재 시 KB 답변 경로로 이동하지 않는다.
구현 저해 문제: 비로그인 위젯 요청 실패로 인해 pending 상태가 실제로 진행되는지 검증할 수 없었다. 정책이 맞더라도 **검증 불가**가 지속되었다.

7. 로깅/디버그 필수 기록
핵심 사항: `INTENT_ROUTED`, `KB_MATCH_COMPLETED`, `KB_LLM_PROMPT_BUILT` 등 이벤트가 항상 기록되어 원인 추적이 가능해야 한다.
구현 저해 문제: 위젯 요청이 실패하는 경우 이벤트 자체가 생성되지 않아 장애 원인을 로그로 추적할 수 없었다. 정상 요청일 때도 `kb_topics`가 재입고로 고정되어 KB 전달 문제를 명확히 드러냈다.

## 2. 로그 기반 근거 요약

- `servicePageLog(bot-20260312-0847).md`, `servicePageLog(bot-20260312-0911).md`에서 사용자 질문 “신청 시기”는 `KB_MATCH_COMPLETED matched=false`, `KB_LLM_PROMPT_BUILT reason=KB_NO_MATCH`, `kb_topics=["재입고"]`, `kb_context_size=43`로 기록되었다.
- 최종 응답은 “KB에 해당 정보가 없습니다… - 재입고” 형태로, 실제 KB 내용과 불일치한다.
- 2026-03-12 `/embed` 테스트에서 `inline_kb`에 `kb.txt` 전체가 포함된 요청은 `KB_MATCH_COMPLETED matched=true`, `KB_LLM_PROMPT_BUILT kb_context_size=1077`, `evidence_keys` 생성이 확인되었다. 즉 런타임은 정상이며 **입력 전달 실패가 원인**이다.
- 2026-03-12 `http://localhost:3000/` 위젯 테스트에서 “볼캡 재입고”는 재입고 흐름으로 정상 응답되었고, 요청 payload에 재입고 샘플 `inline_kb`가 포함되었다.

## 3. 해결이 어려워진 원인

1. 동일 기능이 `/embed`, `/app/laboratory`, 홈 위젯 등 여러 진입 경로로 분산되어 있어, 입력 전달 경로가 일관되지 않았다.
2. 정책 입력 탭과 대화 탭의 상태가 분리되어 있고, 입력값 동기화가 누락되면 `inline_kb`가 전달되지 않는다.
3. 샘플 KB가 기본값으로 유지되어 “KB가 전달됐다”는 착시를 만들었고, 실제 전달된 KB는 로그로만 확인 가능했다.
4. 요청 실패/401 등으로 이벤트 로그가 누락되는 구간이 있어 원인 추적이 지연되었다.
