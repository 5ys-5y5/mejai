# 답변
아래는 3번(API 계약서 초안)을 어떻게 진행할지에 대한 상세 설명입니다.

1) 목적
- 프론트와 백엔드가 같은 규격으로 통신하도록 API 규격(계약서)을 먼저 고정
- 구현 전에 필수 필드/상태코드/에러 규칙을 합의하여 개발 리스크를 줄임

2) 기준
- docs/mejai_Guideline.md (F-B1 필수 로그/요약/확인/응답/이관/만족도)
- 현재 Supabase 테이블 스키마 (sessions, turns, review_queue, event_logs, audio_segments, knowledge_base, organizations, audit_logs)

3) 범위(1차 MVP 계약)
- 인증: 이메일 가입/로그인(확인된 계정만 허용)
- 세션 목록/상세
- 턴 로그(ASR/요약/확인/최종응답/만족도)
- 오디오 구간
- 이벤트 로그
- 리뷰 큐
- KB 문서
- 감사 로그

4) 산출물(문서 구조)
- docs/api_contract.md 로 생성
- 섹션 구성:
  1) 공통 규칙 (base URL, auth header, pagination, sorting)
  2) 에러 규격 (code/message/trace_id)
  3) 엔드포인트 목록 (method/path/설명)
  4) 요청/응답 스키마 (JSON 예시)
  5) 화면별 매핑 (어떤 UI가 어떤 API를 사용)

5) 결과
- 프론트는 이 문서를 기준으로 API 호출 로직을 구현
- 백엔드는 동일 문서를 기준으로 실제 API 제작
